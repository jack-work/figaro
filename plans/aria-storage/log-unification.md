## Core conceptual frame: tics and agent messages

A **tic** is the atomic unit of input between LLM round-trips. It collects everything that happened between the previous assistant message and the next thing figaro sends to the LLM:

- The user's text (if a user prompted)
- `tool_result` content blocks (if the previous assistant turn called tools)
- Patches (chalkboard mutations: client-supplied via `figaro.prompt`, or generated by reactive triggers)

An **agent message** is the assistant's response ŌĆö what the LLM produces in reply to one tic.

The IR is a strict alternation of tic / agent message in the typical case, though the IR layer **does not enforce alternation**; the projection enforces whatever wire alternation a given provider needs (Anthropic does, others may not).

```
tic_0 ŌåÆ agent_0 ŌåÆ tic_1 ŌåÆ agent_1 ŌåÆ tic_2 ŌåÆ agent_2 ...
```

Per the user-confirmed framing: "the singular atomic tic or event encompasses the full time delta between the most recent agent message and the current."

In code, "tic" and "agent message" are both just `message.Message` values. The difference is encoded by Role (`RoleUser` for tics, `RoleAssistant` for agent messages) and the content blocks they carry. **Naming-wise we keep `RoleUser` / `RoleAssistant`** for direct wire-affinity ŌĆö saves a layer of translation.

## Why this exists

Inspection of an existing aria after the cache-control work landed surfaced two observable issues plus a deeper architectural one:

1. **Chalkboard patches don't appear in the persisted aria log.** System-reminder content the model saw at request-time was added at projection time and discarded after the HTTP call. The IR was missing what the model actually saw.
2. **Logical-time collisions** between the message log and the chalkboard log (separate counters, both starting at 1).
3. **The IR / wire shape were diverging more than necessary.** Tool results lived as separate IR messages and got batched at projection (`pendingToolResults`); patches lived in a parallel store and got injected at projection. The wire format expects coherent alternating user/assistant messages with content blocks; the IR was a sparse pre-batched thing the projection had to reconstruct.

The v3 design folds patches into Message and (in Stage D) folds tool_result blocks into the tic Message they belong to. The IR ends up structurally close to wire ŌĆö one Message of Role: user with text + tool_result blocks + patch-derived blocks corresponds 1:1 to one wire user message ŌĆö without the projection doing reconstruction work.

## Design

### Single IR type ŌĆö Message

```go
// internal/message/message.go (proposed v3)

type Message struct {
    LogicalTime uint64    `json:"lt"`
    Timestamp   int64     `json:"ts"`

    Role    Role      `json:"role"`
    Content []Content `json:"content"`

    // Chalkboard mutations that arrived during this tic.
    // Populated only on user-role messages, typically.
    Patches []Patch `json:"patches,omitempty"`

    // Assistant-only metadata.
    Model      string     `json:"model,omitempty"`
    Provider   string     `json:"provider,omitempty"`
    Usage      *Usage     `json:"usage,omitempty"`
    StopReason StopReason `json:"stop_reason,omitempty"`

    // Per-provider wire-format cache.
    Baggage Baggage `json:"baggage,omitempty"`
}
```

Notes:
- `LogicalTime` and `Timestamp` already existed on Message. Stay where they are.
- Roles stay as `RoleUser` / `RoleAssistant` (`RoleSystem` for the deprecated Header). `RoleToolResult` retires ŌĆö tool result data becomes content blocks of type `tool_result` inside a `RoleUser` message. See ┬¦ ContentToolResult below.
- `Patches []Patch` is new. A user-role Message with non-empty Patches represents a tic that brought chalkboard state changes.
- `Baggage` changes type ŌĆö from the old `map[string]json.RawMessage` (single blob per provider) to the new variadic-per-provider shape. See ┬¦ Baggage.
- The existing scalar `ToolCallID` / `ToolName` Message-level fields **retire**; they only made sense for the per-result-message-per-tool model. Their data moves into per-content-block `tool_call_id` fields. See ┬¦ ContentToolResult.

There is no `LogEntry` type. The aria's timeline is `[]Message`, period.

### `Block` simplifies

```go
type Block struct {
    Header   *Message  // (still deprecated; system prompt ŌåÆ chalkboard.system.prompt in Stage C)
    Messages []Message
}
```

`Block.Messages` keeps its name (post-revert it was never renamed). Semantically, it's now the timeline log; each Message is either a tic or an agent message.

### Tool results as content blocks

A new `ContentType` value:

```go
const (
    ContentText        ContentType = "text"
    ContentImage       ContentType = "image"
    ContentThinking    ContentType = "thinking"
    ContentToolCall    ContentType = "tool_call"   // existing ŌĆö assistant emits these
    ContentToolResult  ContentType = "tool_result" // NEW ŌĆö user message carries these
)
```

`Content.ToolCallID` (already exists) ties a tool_result block to the assistant tool_call it answers. The existing `Content.IsError` field (for tool result error flag) stays where it is.

Today's `Message.ToolCallID` and `Message.ToolName` (Message-level scalar fields, populated when `Role == RoleToolResult`) retire. The per-tool data moves down into per-block fields. One user-role Message can carry one Content block per tool result, plus any text the user typed.

### `Patch` moves to the message package

The IR home for `Patch` is `internal/message/patch.go`. The `chalkboard` package re-exports it via type alias for callers that operate on chalkboard's machinery (Snapshot.Diff/Apply/Merge, etc.).

```go
// internal/message/patch.go
type Patch struct {
    Set    map[string]json.RawMessage `json:"set,omitempty"`
    Remove []string                   `json:"remove,omitempty"`
}

func (p Patch) IsEmpty() bool { ... }
```

```go
// internal/chalkboard/chalkboard.go
type Patch = message.Patch  // alias, methods accessible via either name
```

Methods that depend on `Snapshot` (currently `(p Patch).Entries(prev Snapshot)`) live in chalkboard as package-level functions or methods on a chalkboard-side wrapper. The plain delta type lives in `message`.

### Baggage ŌĆö variadic, per-provider, with fingerprint

```go
// internal/message/baggage.go (NEW)

type Baggage struct {
    Entries map[string]ProviderBaggage `json:"entries,omitempty"`
}

type ProviderBaggage struct {
    // Wire-format messages this Message materializes for the provider.
    // Length is typically 1 (one IR Message ŌåÆ one wire message). May be
    // 0 (state-only tics that produce no wire output ŌĆö bootstrap,
    // rehydrate). May be >1 for renderers that fan out (e.g.
    // tool-injection rendering of patches).
    Messages []json.RawMessage `json:"messages"`

    // Renderer-config fingerprint at write time. Stale entries log a
    // warning on read but do not auto-invalidate; force-rerender is
    // an explicit operation.
    Fingerprint string `json:"fp,omitempty"`
}
```

Migration: existing on-disk Message.Baggage is `map[string]json.RawMessage` keyed by provider name. v3 reads this old shape on load and converts to `ProviderBaggage{Messages: []json.RawMessage{<old blob>}}` (single-element list, no fingerprint). Writes use the new shape exclusively.

### State-only tics

Bootstrap and rehydrate produce user-role Messages with **only Patches** ŌĆö no Content. They emit no wire output (their `ProviderBaggage.Messages` is empty after rendering); their effect is entirely to update the chalkboard snapshot the projection reads when computing the next *real* send.

```go
// Aria bootstrap message
Message{
    Role:    RoleUser,
    LogicalTime: 1,
    Patches: []Patch{{Set: {
        "system.prompt":            <credo>,
        "system.skills":            <skills>,
        "system.skills_digest":     ...,
        "system.credo_digest":      ...,
        "system.model":             ...,
        "system.reminder_renderer": ...,
        "system.max_tokens":        ...,
    }}},
    // Content is empty.
}
```

The IR shape for an aria with bootstrap + first prompt + first reply:

```
Message{Role: user, lt: 1, Patches: [bootstrap]}              ŌåÉ state-only tic; emits no wire output
Message{Role: user, lt: 2, Content: [text("hello")], ...}     ŌåÉ real first tic
Message{Role: assistant, lt: 3, Content: [text("hi")], ...}   ŌåÉ agent_0
```

Subsequent rehydrate fits the same shape ŌĆö a state-only tic between two real-content tics.

### IR alternation is *not* enforced

Per user-confirmed direction: the IR doesn't constrain order. A provider may require alternation on the wire (Anthropic does); the projection layer is responsible for satisfying the wire's needs. The IR can have consecutive user-role Messages (e.g. a state-only tic immediately before a real tic, two state-only tics back-to-back) without internal complaint.

Ramification: providers with strict alternation handle "shadow" state-only tics by emitting empty wire output for them. Providers without strict alternation (a future possibility) can emit them however they like.

### Agent loop: the in-progress tic

The actor loop in `internal/figaro/agent.go` accumulates events into an **in-progress user-role Message** that's mutable until finalized at send-time:

```
state: inProgressTic *Message  // mutable; nil between turns

on eventUserPrompt(text, chalkboardInput):
    if inProgressTic == nil: inProgressTic = &Message{Role: user}
    inProgressTic.Content = append(inProgressTic.Content, textContent(text))
    if patch := computePatchFromInput(chalkboardInput); !patch.IsEmpty():
        inProgressTic.Patches = append(inProgressTic.Patches, patch)
    finalizeAndSend()

on eventToolResult(callID, content, isErr):
    if inProgressTic == nil: inProgressTic = &Message{Role: user}
    inProgressTic.Content = append(inProgressTic.Content, toolResultContent(callID, content, isErr))
    pendingTools--
    if pendingTools == 0 && pendingToolCalls is empty: finalizeAndSend()

on eventLLMDone(assistantMessage):
    memStore.Append(assistantMessage)  // agent message, immutable on append
    inProgressTic = nil
    if assistantMessage.has tool_uses: schedule tool execution
    else: turn complete; await next user prompt

on eventRehydrate():  // future control RPC
    if inProgressTic == nil: inProgressTic = &Message{Role: user}
    inProgressTic.Patches = append(inProgressTic.Patches, rehydrateDiff)
    // does NOT finalizeAndSend ŌĆö control plane only

on finalizeAndSend():
    inProgressTic.LogicalTime = memStore.AllocLT()
    inProgressTic.Timestamp = now()
    memStore.Append(*inProgressTic)
    inProgressTic = nil
    startLLMStream()
```

Once `memStore.Append` is called for the in-progress tic, it's immutable IR. Pre-finalize, it's a buffer. The actor's single goroutine model means no concurrent access.

Bootstrap is a special-case finalize: at agent construction, build the bootstrap tic, append, but **do not** start an LLM stream. The first real user prompt assembles its own (non-bootstrap) tic.

### Projection at the provider

The renderer becomes a pure function on `*Message`:

```go
// Conceptual signature; reads only self + prior messages + chalkboard.
type Renderer func(self *Message, prior []Message, snapshot chalkboard.Snapshot) []json.RawMessage
```

Concretely for Anthropic:

```go
for _, msg := range block.Messages {
    if cached, ok := msg.Baggage.Entries[providerName]; ok && fingerprintMatches(cached) {
        appendCached(cached.Messages...)
        continue
    }
    wire := renderMessage(&msg, prior, snapshot)  // tag or tool style
    msg.Baggage.Entries[providerName] = ProviderBaggage{Messages: wire, Fingerprint: fp}
    appendCached(wire...)
}
```

`prior` is the slice of Messages preceding `msg` in the block. Causal masking ŌĆö the handler can read prior context but not future. For Anthropic specifically, the handler typically needs only `self` and the current snapshot; prior is available for any future need (e.g. a renderer that wants to suppress redundant content).

`pendingToolResults` and `flushToolResults` from current code retire. Tool results are already in their final shape in the IR ŌĆö one user Message with multiple `tool_result` content blocks. Projection emits one wire user message; no batching reconstruction needed.

## Bugs surfaced during implementation

- **Stage E cache-lookup admitted non-message-shaped translation entries.** The Stage D.2f write-through persistence introduced translation log entries for the system block array, keyed by the bootstrap flt. The cache-hit path in `projectMessages` used `json.Unmarshal` to decode any cached entry as a `nativeMessage`. Lax JSON decoding turned the `systemBlock`-shaped bytes (`{type, text}`) into `nativeMessage{Role: "", Content: nil}`, which the Anthropic API rejects with `messages.0.role: Input should be 'user', 'assistant'`. Fixed in `b7db923` (Stage E commit) by validating `cached.Role` before treating an entry as a usable per-message projection. Regression test in `internal/provider/anthropic/projection_test.go`. **Lesson:** any future translation-log entry kind that doesn't have a "role" field needs the same validation ŌĆö or, better, a `Kind` discriminator on `TranslationEntry` so the per-message lookup path doesn't even consider non-message entries. Deferred; for now the role-validation fix is sufficient.

## Outstanding work

The prefix-byte-stability regression is the single most important loose end. The `cache_control` invariant (#11 in `agents.md`) depends on byte-equality of the prefix (system blocks + tools + messages[0..N-2]) across requests within an aria's lifetime. Without a regression test covering bootstrap + chalkboard mutations + tool result tics + the new skills system block, a future change can silently degrade Anthropic's cache_read tokens ŌĆö the runtime cost shows up as bills, not as test failures. Restoration is straightforward: a new test file that builds a Block, projects it, mutates the chalkboard, projects again, and asserts byte-equality of `req.System ++ req.Tools ++ req.Messages[:len-1]`.

## Resolved decisions (v3 summary)

- **One IR primitive ŌĆö `Message`.** No `LogEntry` wrapper. Tics and agent messages are both Messages, distinguished by Role.
- **Patches as a field on Message** (`Patches []Patch`), populated for user-role messages that arrived with chalkboard mutations.
- **Tool results as content blocks**, not separate Messages. `RoleToolResult` retires; `ContentToolResult` content type added with per-block `tool_call_id`.
- **State-only tics** (Patch-only Messages) are valid IR; emit zero wire output but contribute to the chalkboard snapshot the projection reads.
- **`Baggage` is variadic per provider** with a renderer-config fingerprint; cache is sticky.
- **No alternation invariant in IR.** Provider's projection enforces wire alternation.
- **Agent loop builds an in-progress tic** mutably; finalizes (memStore.Append + send) when ready.
- **`Patch` lives in `internal/message/`** canonically; `chalkboard` aliases it.
- **`chalkboard.State` per-aria handle.** Snapshot returns a copy; comment in code marks the path to immutable / lock-guarded data structures for future parallelism.
- **Storage layout `arias/{id}/aria.jsonl` + `arias/{id}/chalkboard.json`**, lazy NDJSON via rewrite-tmp-rename.
- **Scribe runs only at bootstrap and rehydrate.** Subsequent turns read system prompt from chalkboard.

## Remaining open questions

1. **`Block.Header` retirement.** Stage C moves the system prompt into chalkboard. Block.Header becomes unused. Probably delete it entirely. Worth confirming there are no late-stage uses we're forgetting.
2. **Renderer interface formalization.** Today renderers are package-private functions in the Anthropic package. Stage D might formalize an interface for cross-provider reuse. Or might not ŌĆö let the Anthropic implementation drive what's worth abstracting.
3. **Cache-busting CLI command.** `figaro chalkboard rerender` to force-invalidate Baggage entries for the active provider. Trivial follow-up; not blocking.
4. **The `Patches` field on assistant messages?** Today the design says patches live on user-role messages (tics). But assistant messages could in principle carry chalkboard mutations too ŌĆö e.g. an assistant turn that updates `system.last_intent` based on the response. Punt: keep `Patches` populated only on user-role messages for v1; if a use case appears, extend.
